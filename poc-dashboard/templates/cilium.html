Cilium Network Policy

eBPF-powered Kubernetes networking — enforce identity-aware network policies between workloads at the kernel level.

WHAT IS CILIUM?

Cilium is the networking and security layer built into every node of this Kubernetes cluster. It uses eBPF — a Linux kernel technology — to enforce network policies at the kernel level, before packets even reach the application.

Every pod in this cluster runs with Cilium active. Out of the box, Cilium's default behavior is allow all: pods can be reached by any device on the network and can freely communicate with each other.

The goal of this demo is workload microsegmentation — applying policies that isolate individual workloads so that even if one pod is compromised, an attacker cannot move laterally to other services in the cluster, and users can only reach applications through the controlled Secure Access path.

Allow All Traffic

{% if not zero_trust_active %} ACTIVE {% endif %}

Removes all Cilium network policies from the piap namespace. Cilium's default with no policies is allow all — pods can be reached from the local LAN, from the Resource Connector, and from each other without restriction.

What gets allowed:
  • Direct LAN access to all pods (SSH, RDP, web services, Splunk, …)
  • Traffic between pods within the cluster
  • Traffic from the Cisco Resource Connector to all pods

Zero Trust

{% if zero_trust_active %} ACTIVE {% endif %}

Applies a CiliumNetworkPolicy that enforces Zero Trust at the network layer. All workload pods accept ingress only from the Resource Connector — users must authenticate through Cisco Secure Access to reach any service.

Policy: piap-zero-trust
  • Ingress: only from the Resource Connector (cilium_host IP, Docker bridge, and host entity)
  • Egress: unrestricted (responses & outbound traffic still flow)
  • Services switched to externalTrafficPolicy: Local to preserve real client IPs
ℹ️ Hubble UI and the PoC Dashboard remain reachable.
